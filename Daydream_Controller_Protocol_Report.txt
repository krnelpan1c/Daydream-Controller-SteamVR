Daydream Controller Protocol & Communication Report
===================================================

This report details the operational protocol and implementation specifics of the Google Daydream Remote based on reverse engineering and driver development.

1. BLE CONNECTION & PAIRING
---------------------------
- **Pairing Bypass (Error Code 19):** Windows often struggles to pair with the Daydream remote natively using default flows. It must be paired dynamically utilizing the Windows 'Custom' pairing handler set exclusively to 'ConfirmOnly'.
- **Service UUID:** `0000fe55-0000-1000-8000-00805f9b34fb`
- **Characteristic UUID:** `00000001-1000-1000-8000-00805f9b34fb`
- Data is received via GATT characteristic notifications triggered asynchronously.

2. PACKET STRUCTURE
-------------------
The controller sends data in 20-byte, bit-packed payload structures. Due to the high density of sensor data, the values are NOT byte-aligned and must be extracted dynamically via bit-shifting based on precise bit offsets.

The absolute bit boundaries inside the 160-bit (20 byte) payload are as follows:
- Timestamp: 9 bits (0 -> 8)
- Sequence: 5 bits (9 -> 13)
- Orientation X: 13 signed bits (14 -> 26)
- Orientation Y: 13 signed bits (27 -> 39)
- Orientation Z: 13 signed bits (40 -> 52)
- Accelerometer X: 13 signed bits (53 -> 65)
- Accelerometer Y: 13 signed bits (66 -> 78)
- Accelerometer Z: 13 signed bits (79 -> 91)
- Gyroscope X: 13 signed bits (92 -> 104)
- Gyroscope Y: 13 signed bits (105 -> 117)
- Gyroscope Z: 13 signed bits (118 -> 130)
- Touchpad X: 8 unsigned bits (131 -> 138)
- Touchpad Y: 8 unsigned bits (139 -> 146)
- Buttons: 5 individual bits (147 -> 151)

3. TOUCHPAD & BUTTONS
---------------------
To read the trackpad, unsigned 8-bit integers (0-255) must be extracted from the bounds listed above. Both X and Y will report 0 when the pad is untouched.
- Daydream Pad X: 0 is the far left, 255 is the far right.
- Daydream Pad Y: 0 is the top, 255 is the bottom.

The 5 specific button flags reside precisely across the lower 5 bits of Byte 18.
- Bit 151 (Byte 18 & 0x01) -> Touchpad Click
- Bit 150 (Byte 18 & 0x02) -> Home Button
- Bit 149 (Byte 18 & 0x04) -> App Button (Minus)
- Bit 148 (Byte 18 & 0x08) -> Volume Down
- Bit 147 (Byte 18 & 0x10) -> Volume Up

4. SPATIAL TRACKING (IMU ORIENTATION)
-------------------------------------
The Daydream controller operates exclusively as a 3DoF (Degrees of Freedom) tracking wand. It calculates the internal orientation of its chassis referencing real world gravity using a fusion filter from its internal IMU.

**CRITICAL: Axis-Angle Notation**
The `oriX`, `oriY`, and `oriZ` variables transmitted inside the BLE packet are NOT Euler tracking angles, and passing them through Euler rotational functions will cause the resulting transform to spiral sideways and pitch backwards. 
Instead, they are the individual factors of an **Axis-Angle specific vector**.
To convert this to a usable spatial quaternion:
- The scalar angle representing the rotation depth is equivalent to the magnitude of the vector: `Angle = sqrt(x^2 + y^2 + z^2)`
- The quaternion is built by assigning `W = cos(Angle / 2)` and then scaling the respective component axes by `sin(Angle / 2) / Angle`.

**Coordinate Systems:**
The base axis-angle outputs generated by the Daydream controller are strictly compliant with the OpenGL Standard Right-Hand rule natively, meaning permutations are not needed to match platforms operating within standard 3D coordinate planes (such as OpenVR / SteamVR).
- `oriX`: Handles physical Pitch (X-Axis rotation).
- `oriY`: Handles physical Yaw (Y-Axis rotation).
- `oriZ`: Handles physical Roll / Twisting (Z-Axis rotation).

Because absolute 3DoF positioning contains inherent drift around the Y axis over extended bounds of time, any integration schema requires a manual zeroing step. In order to anchor the local coordinate space to the player's world horizon without destroying absolute gravity orientation, you merely resolve the differential between the controller's immediate relative Yaw vector and the headset's physical Yaw vector via explicit quaternion rotation logic.
